Advice

Descriptive Variable Names
When writing programs, you may have named variables like 'x', 'a', or 'i'. And why shouldn't you? The compiler doesn't complain and your program works fine with them. But, a principle you should keep in mind is that code read more often than it is written. Now, put yourself in the shoes of someone who has to read your code. What does 'x' mean? Why does 'x' hold this value and not 'y'? It is natural for such questions to arise in the mind of a reader. To eliminate them, make sure that you name variables in such a way that their purpose can be understood from their name. For example, if a variable is supposed to hold the area of a rectangle, a name like 'rect_area' is more appropriate than 'x' or 'y'. The principle can be extended to other things like functions and classes. For a function to calculate the area of a rectangle, the name 'area_of_rect' is more appropriate than 'func'. Don't be lazy and make your variable names descriptive.

Don't repeat yourself (DRY)
When writing code, you may have experienced a situation where you end up writing the same code in multiple places. For example, when you need to calculate the absolute value of a number in C without the use of library functions, you may write a simple ternary operation ( (a < 0)? -a : a; ) wherever the value is required. If you have done something similar, you've violated the principle of DNRY, which says that you should not repeat code and instead, reuse it whenever possible. The problems from not following this principle are obvious. You'll need to make the same change to the code at multiple places if you make a change at one place. To avoid this, pay attention to the code you're writing and refactor the repeated code as a function or some other form of abstraction wherever possible.
