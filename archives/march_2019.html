Descriptive Variable Names
It is easy to write a software that is easy to implement and works satisfactorily. But it is very hard to write reliable, understandable, and maintainable code. One important aspect is using good variable and function names. Variable names must be - understandable, descriptive, and pronounceable. Some universal rules of naming are as follows : - 1. Classes must begin with an uppercase 2. Variables, functions, constructors, etc must begin with a lowercase If your code has variable names like a, b, c, d, aa, bb, cc, and so on, after a certain amount of lines it gets difficult to track what variable is storing what. But, if your code has variable names like prin_amt, time, roi, and si, then it gets easier for the coder to understand what variable stores the principal amount, the time, the rate of interest, and the simple interest. But often times, it has been seen the more descriptive your variable names get, the tougher it becomes to type. So, to have a neater and cleaner and a easily navigable code, name your variables to-the-point.

Do not repeat yourself (DNRY)
When writing code, you may have experienced a situation where you end up writing the same code in multiple places. For example, when you need to calculate the absolute value of a number in C without the use of library functions, you may write a simple ternary operation ( (a < 0)? -a : a; ) wherever the value is required. If you have done something similar, you've violated the principle of DNRY, which says that you should not repeat code and instead, reuse it whenever possible. The problems from not following this principle are obvious. You'll need to make the same change to the code at multiple places if you make a change at one place. To avoid this, pay attention to the code you're writing and refactor the repeated code as a function or some other form of abstraction wherever possible.
